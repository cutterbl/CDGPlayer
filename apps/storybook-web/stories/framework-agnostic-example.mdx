# Framework-agnostic Implementation Guide

This guide lives next to the framework-agnostic demo story so implementation notes and runnable behavior stay together.

Source of truth for this guide: `apps/demo/src/app/app.element.ts`.

## Step 1: Create Player and Controls Model

```ts
import { createPlayer } from '@cxing/cdg-player';
import {
  createControlsModel,
  createPlayPauseControl,
  createCurrentTimeDisplay,
  createProgressControl,
  createDurationDisplay,
  createVolumeControl,
  createTempoControl,
  createKeyControl,
} from '@cxing/cdg-controls';

const player = createPlayer({
  options: {
    canvas,
    audio,
  },
});

const model = createControlsModel({
  options: { player },
});

createPlayPauseControl({ container: transport, model });
createCurrentTimeDisplay({ container: transport, model });
createProgressControl({ container: transport, model });
createDurationDisplay({ container: transport, model });

createVolumeControl({
  container: settings,
  model,
  orientation: 'vertical',
});
createTempoControl({
  container: settings,
  model,
  orientation: 'vertical',
});
createKeyControl({ container: settings, model });
```

`Key` uses a vertical range slider with datalist labels shown as half-step values for singer-facing clarity (for example `.5`, `1`, `1.5`), but control writes still use integer semitone values internally.

## Step 2: Subscribe to Model Updates

```ts
let lastStatus: ControlsViewState['status'] | null = null;

const unsubscribe = model.subscribe((state) => {
  if (state.status !== lastStatus) {
    setStatusMessage(`Status: ${state.status}`);
    lastStatus = state.status;
  }

  syncTitleImage(state.status);
  syncLayout(state.status);
});

setStatusMessage('Choose a track to start.');
syncTitleImage('idle');
syncLayout('idle');
```

This is the same pattern used by the framework-agnostic demo custom element: the UI reacts to model state changes, not to direct control events.

## Step 3: Load a User-selected Track

```ts
fileInput.addEventListener('change', () => {
  const selectedFile = fileInput.files?.item(0);
  if (!selectedFile) {
    return;
  }

  player.stop();
  setTitleMetadata(null);
  syncTitleImage('loading');
  syncLayout('loading');
  setStatusMessage('Loading track...');

  void player
    .load({
      input: { kind: 'file', file: selectedFile },
    })
    .then((loadedTrack) => {
      const metadata = loadedTrack?.metadata;
      if (metadata) {
        setTitleMetadata({
          title: metadata.title,
          artist: metadata.artist,
        });
      }

      syncTitleImage('ready');
      setStatusMessage('Track loaded.');
    })
    .catch((errorValue: unknown) => {
      const message =
        errorValue instanceof Error ? errorValue.message : 'Unknown load error';
      setStatusMessage(`Load failed: ${message}`);
      syncTitleImage('error');
      syncLayout('error');
    })
    .finally(() => {
      fileInput.value = '';
    });
});
```

For picker UX, configure the file input as `accept="audio/*,.zip,application/zip"`.
When `loadedTrack.hasGraphics` is `false`, keep transport controls active and hide the canvas region because the track is audio-only.

stage.addEventListener('click', (event) => {

## Step 5: Teardown

```ts
const parts = [
  createPlayPauseControl({ container: transport, model }),
  createCurrentTimeDisplay({ container: transport, model }),
  createProgressControl({ container: transport, model }),
  createDurationDisplay({ container: transport, model }),
  createVolumeControl({ container: settings, model, orientation: 'vertical' }),
  createTempoControl({ container: settings, model, orientation: 'vertical' }),
  createKeyControl({ container: settings, model }),
];

const dispose = () => {
  unsubscribe();
  for (const part of parts) {
    part.dispose();
  }
  model.dispose();
  player.dispose();
};
```

## Common Mistakes

- Mounting all controls to one container when UI is split across transport/settings regions.
- Updating status/layout from ad-hoc event handlers instead of a single `model.subscribe(...)` path.
- Not resetting file input after a load attempt.
- Forgetting to handle load errors in UI status messaging.
- Leaving controls mounted while replacing the player instance.
