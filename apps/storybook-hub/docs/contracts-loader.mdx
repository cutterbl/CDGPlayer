# Loader Contract

`@cxing/media-loader` accepts different input sources and returns one normalized track payload shape for audio or video playback.

Metadata extraction behavior:

- First tries embedded audio tags (via `jsmediatags`) for fields like title and artist.
- Video payloads skip embedded tag extraction and rely on filename fallback metadata.
- Falls back to filename parsing when tags are missing or unreadable.

Use loader when:

- you want to load browser-supported media files and karaoke zip bundles
- you want a consistent parsed payload before passing to player
- you want metadata (`title`, `artist`, etc.) along with media bytes

## Browser Codec Support

- Best compatibility across Chrome, Firefox, and Safari is `video/mp4` with H.264 video and AAC audio.
- `video/webm` is usually fine in Chrome and Firefox, but Safari support varies by codec/profile and OS version.
- Container extensions like `.avi`, `.mkv`, and `.mov` may be accepted by the picker, but playback still depends on the actual codec streams inside the file.
- The loader uses browser `canPlayType(...)` checks as an initial filter only; runtime decode validation still happens in the player because some browsers report `maybe` for files whose video stream they cannot actually decode.

## Logging Behavior

- Loader internals emit scope-prefixed logs (for example `[media-loader-core]`, `[media-loader]`, `[media-loader-worker]`).
- `options.debug` enables debug trace output only.
- `info()` and `error()` events are not debug-gated:
  - metadata filename fallback is logged at `info`
  - load/worker failures are logged at `error`

## Minimal Usage

```ts
import { createLoader } from '@cxing/media-loader';

const loader = createLoader();
const loadedTrack = await loader.load({
  input: { kind: 'file', file: selectedFile },
  options: { debug: true },
});
```

## Output Shape You Can Rely On

```ts
// same shape for framework-agnostic and framework integrations
loadedTrack.metadata.title;
loadedTrack.audioBuffer;
loadedTrack.mediaKind;
loadedTrack.mediaMimeType;
loadedTrack.audioMimeType;
loadedTrack.hasGraphics;
loadedTrack.cdgBytes;
```

`audioMimeType` is retained for backward compatibility. New integrations should prefer `mediaKind` and `mediaMimeType`.

### Utility Function Guidance

- `classifyMediaKind(...)` prioritizes MIME and then falls back to extension-based classification.
- `canBrowserPlayMedia(...)` is a capability signal, not a final decode guarantee.
- `inferMimeTypeFromExtension(...)` should be treated as best-effort mapping, especially for container formats.

## Common Mistakes

- Treating `hasGraphics` as always true; audio-only tracks return `hasGraphics === false` with `cdgBytes === null`.
- Assuming a supported container means a supported codec; `.mp4` is only broadly safe when the video stream is H.264 and the audio stream is AAC.
- Assuming video tracks preserve `.cdg` graphics pairing from zip bundles; when video is selected as the primary payload, CDG graphics are ignored.
- Assuming metadata fields are always populated; handle empty strings safely because some files have no tags and no parseable filename pattern.
- Re-loading repeatedly without cancellation logic in your app flow.
