# Contribution Guidelines

This page explains how to contribute changes through a pull request.

## Contribution Flow

Contributions should be made as a pull request.

Before opening or updating a PR, run the core validation checks locally:

```bash
pnpm run ci:validate
pnpm run ci:test
pnpm run ci:build
pnpm run ci:docs
```

## Code Structure Expectations

- Keep package boundaries intact: loader, player, controls, demos, and Storybook docs should stay in their respective scopes.
- Prefer small, focused changes instead of broad refactors unless the task requires structural work.
- Keep implementation guides and example stories aligned with the code they document.
- Update documentation whenever public behavior, integration flow, or architecture guidance changes.

## TypeScript and ECMAScript Standards

- Use TypeScript for shipped application and package code.
- Keep types explicit and maintain strict typing across public APIs and internal state transitions.
- Use modern ECMAScript patterns already established in the repo: ESM modules, async/await, named-argument objects, and standard web platform APIs.
- Avoid reintroducing legacy patterns when a modern TypeScript or browser-native approach already exists in the codebase.

## Documentation Workflow

- Keep runtime contracts documented as MDX in `apps/storybook-hub/docs`
- Use cross-reference links to connect loader/player/controls and example pages
- Keep both integration styles represented in code examples

## Local Performance Monitoring And Reporting

Performance monitoring and reporting are local-development workflows only.
They are not required CI gates and are not part of the published `@cxing/*` runtime APIs.

Use these commands from repository root:

```bash
pnpm run perf:check-readiness
pnpm run perf:generate-artifact
pnpm run perf:check-artifacts
```

### What Each Command Does

- `perf:check-readiness`: verifies required worker/render paths and perf budget config are present.
- `perf:generate-artifact`: aggregates local capture files into `artifacts/perf/localdev-aggregated.json`.
- `perf:check-artifacts`: validates `artifacts/perf/localdev-aggregated.json` against configured budgets.

### Benchmark Budgets

Budgets are configured in `configs/perf-budgets.json`:

- Main-thread p95 frame CPU must be <= `20ms`.
- Worker p95 frame CPU must be <= `8ms`.
- Worker max transferred bytes must be <= `307200`.

### How To Read Results

- **p95 frame CPU**: near worst-case frame processing cost; useful for detecting tail-latency spikes.
- **max transferred bytes**: highest per-frame copy volume for worker dispatch; useful for spotting large transfer regressions.
- Treat a single run as directional only; compare multiple local runs when evaluating regressions.

## Contributor Checklist

1. Update docs when API behavior changes.
2. Keep examples aligned with current recommended usage.
3. Keep new code consistent with package boundaries and existing structure.
4. Prefer TypeScript and modern ECMAScript patterns over legacy alternatives.
5. Run CI checks locally before handoff.
